A schema is defined using a `SchemaFactory` object, constructed with a unique string (such as a UUID).
`SchemaFactory` provides five leaf types — `boolean`, `string`, `number`, `null`, and `handle` — and three methods for internal nodes: `object()`, `array()`, and `map()`.
See [defining a schema](../../start/tree-start.mdx#defining-a-schema) for an example.

:::note
Once you have documents using your schema, see [Schema Evolution](./schema-evolution/index.mdx) for guidance on safely updating your schema over time.
:::

## Object Schema

Use `object()` to define a schema. The method returns a schema object that you extend to create a TypeScript class. In this example:

-   `id`, `text`, `author`, and `lastChanged` are leaf node properties.
-   `votes` is an inline array node whose items are strings.

```typescript
const noteSchema = sf.object("Note", {
	id: sf.string,
	text: sf.string,
	author: sf.string,
	lastChanged: sf.number,
	votes: sf.array(sf.string),
});
```

Create a TypeScript class by extending the schema object:

```typescript
class Note extends noteSchema {
	/* class members */
}
```

You can also inline the `object()` call (this style is used throughout the rest of this article):

```typescript
class Note extends sf.object("Note", {
	id: sf.string,
	text: sf.string,
	author: sf.string,
	lastChanged: sf.number,
	votes: sf.array(sf.string),
}) {
	/* class members */
}
```

You can add fields, properties, and methods like any TypeScript class.
Methods that write to shared properties are reflected on all clients. For example:

```typescript
public updateText(text: string) {
    this.lastChanged = new Date().getTime();
    this.text = text;
}
```

You can also add members that are local to the current client (not backed by a DDS). For example, a `color` property lets each user set their own note color without affecting other clients:

```typescript
private color: string = "yellow";

public setColor(newColor: string) {
    this.color = newColor;
}
```

:::note

Do _not_ override the constructor of types that you derive from objects returned by the `object()`, `array()`, and `map()` methods of `SchemaFactory`. Doing so has unexpected effects and is not supported.

:::

Create a schema for a group of notes. `Group.notes` is an inline array node — we'll give it a named type in the [Array schema](#array-schema) section.

```typescript
class Group extends sf.object('Group', {
    id: sf.string,
    name: sf.string,
    notes: sf.array(Note),
}) {}
```

## Array Schema

Since `sf.array(Note)` is needed in multiple places, give it a named type. Methods can be added to wrap [array node APIs](./reading-and-editing.mdx#array-node-apis):

```typescript
class Notes extends sf.array("Notes", Note) {
	public newNote(author: string) {
		// implementation omitted.
	}
}
```

Update `Group` to use the named type:

```typescript
class Group extends sf.object('Group', {
    id: sf.string,
    name: sf.string,
    notes: Notes,
}) {}
```

## Root Schema

The root of the app's data can hold two kinds of children: notes inside groups, and notes outside any group. These are defined as `Items`, an array that accepts both types. Pass an array of schema types to `array()` to allow multiple item types:

```typescript
class Items extends sf.array("Items", [Group, Note]) {
	public newNote(author: string) {
		// implementation omitted.
	}

	public newGroup(name: string): Group {
		// implementation omitted.
	}
}
```

Define the root type:

```typescript
class App extends sf.object("App", {
	items: Items,
}) {}
```

Finally, create a configuration object used when creating or loading a `SharedTree`. See [creating a tree](../../start/tree-start.mdx#creating-a-tree):

```typescript
export const appTreeConfiguration = new TreeViewConfiguration({
	// root node schema
	schema: App,
});
```

## Map Schema

Creating a map schema is like creating an array schema, but using the `map()` method. Since map keys are always strings, no key type parameter is needed. For example, to represent bids in an auction app (mapping user names to bid amounts):

```typescript
class AuctionItem extends sf.map('AuctionItem', sf.number) { ... }
```

Like `array()`, `map()` accepts an array of types when values are not all the same type:

```typescript
class MyMapSchema extends sf.map('MyMap', [sf.number, sf.string]) { ... }
```

## Recursive Schema

You can create recursive types (nodes that contain nodes of the same type). Due to TypeScript limitations, this requires the `objectRecursive()`, `arrayRecursive()`, and `mapRecursive()` variants, and incorrect declarations may not produce type errors. Use `ValidateRecursiveSchema` to catch mistakes at compile time:

```typescript
type _check = ValidateRecursiveSchema<typeof myRecursiveType>;
```

## Setting Properties as Optional

Wrap a type in `SchemaFactory.optional()` to make a property optional:

```typescript
class Proposal = sf.object('Proposal', {
    id: sf.string,
    text: sf.optional(sf.string),
});
```

## Best Practices

### Avoid Optional Maps and Arrays

Optional maps and arrays are usually an anti-pattern. Maps and arrays already have an empty state (no entries / `[]`), and for most applications an empty collection is equivalent to a missing one. Make maps and arrays required unless you have a specific reason (e.g., adding them to an existing type).

When a map or array is optional, every read site must check for `undefined`, and every write site must first create the collection if absent before populating it:

```typescript
const map = myNode.myMap;
if (map === undefined) {
  myNode.myMap = new MyMap({});
}

myNode.myMap.set("key", "value");
```

This incurs an extra write and is lossy under concurrent edits: if two clients both create the map concurrently, only one survives. The other map, and any edits made to it, are lost.

### Avoid Redundant Data

Storing the same data in multiple places requires multiple edits to keep it in sync, increases merge conflicts, and adds document invariants that must be maintained.

For example, storing a user's ID both as a map key and as a property on the user is redundant — to retrieve the user from the map, you already know their ID:

```typescript
class User extends sf.object("User", {
  id: sf.string,
}) {}

/** A map from ID to user */
class Users extends sf.map("Users", sf.string, User) {}
```

Redundancy also applies to derived data. For example, a "total score" that is the sum of all users' scores is fully computable from existing data. Unless computing it is prohibitively expensive, calculate it at runtime (and cache in memory if needed) rather than storing it in the document.

### Factor Out Map and Array Value Types

Prefer named classes for types stored in maps and arrays:

```typescript
class MyFoo extends sf.object("MyFoo", { foo: sf.string }) {}

class MyObject extends sf.object("MyObject", {
  listA: sf.array(MyFoo), // Named (preferred)
  listB: sf.array(sf.object({ foo: sf.string })) // Inlined/unnamed
}) {}
```

Both objects above are structurally equivalent, but only `MyFoo` has an explicit name (inlined types receive an auto-generated name). Using explicit names has two benefits:

- **Type control**: When moving elements between arrays, both arrays must contain elements with the same type name. Explicit names let you control which elements can be moved across which arrays, even when they are structurally identical.
- **Smaller `.d.ts` files**: SharedTree schema types can expand significantly in compiled output. Named types allow the compiler to reference them by name rather than inlining their full structure.
