As your application evolves, you may need to modify your [SharedTree](../index.mdx) schema structure.
SharedTree supports schema updates while preserving the ability to open documents created by earlier versions and to collaborate with older clients (see [understanding compatibility](./index.mdx#understanding-compatibility)).
Verify schema changes using a test with [`snapshotSchemaCompatibility`](../../../api/fluid-framework/#snapshotschemacompatibility-function).

:::note
For detailed information about which types of changes are safe versus breaking, see [types of schema changes](./types-of-changes.mdx).
:::

## Stored vs View Schema

There are two distinct schema types:

- **View schema**: Defined in your application code using [`SchemaFactory`](../../../api/fluid-framework/schemafactory-class).
- **Stored schema**: Persisted with the document. Describes what the tree can contain.

When a document is first created, its stored schema is initialized from the view schema and remains immutable until explicitly [upgraded](./index.mdx#schema-upgrade-process).

### Stored Schema Invariants

The stored schema has additional constraints that don't apply to view schemas:

**1. Complete description** — The stored schema fully describes all possible data in the tree. Every node and field is known to it.

**2. Additive only** — The stored schema can only grow. All node types ever introduced remain permanently, even if the view schema stops using them.

**3. Field key stability** — Field keys on object nodes are permanent. A key can never be changed, reused, or repurposed once introduced.

### Why These Constraints?

Documents may persist indefinitely before being opened again, so an application must handle any historical version.

## Schema Compatibility

When opening a document, SharedTree compares the view schema to the stored schema:

- **Compatible**: Document can be opened and used normally.
- **Upgradeable**: Document can be upgraded to match the view schema using [`view.upgradeSchema()`](../../../api/fluid-framework/treeview-interface#upgradeschema-methodsignature).
- **Incompatible**: Document cannot be opened or upgraded with the current view schema.

The schema must be compatible or upgradeable for the application to function. SharedTree throws if the application attempts to access data with an incompatible view schema.

A document's stored schema initially matches the creation-time view schema. As the view schema changes, the two may become backwards compatible, forwards compatible, both, or neither.

### Understanding Compatibility

#### Backwards Compatibility

**Backwards compatible**: the new view schema is compatible with the old stored schema — new app versions can [upgrade](./index.mdx#schema-upgrade-process) existing documents (though upgrading may affect forwards compatibility). If not backwards compatible, existing documents can no longer be loaded.

#### Forwards Compatibility

**Forwards compatible**: the old view schema is compatible with the new stored schema — older app versions can still open upgraded documents. A change that is backwards but not forwards compatible breaks older clients on upgraded documents; use a [staged rollout](./index.mdx#staged-rollouts) to prevent this.

### Schema Change Examples

#### 😎 Chill
| Backwards Compatible | Forwards Compatible | Rollout Process |
|-----------------------|----------------------|-----------------|
| ✅                    | ✅                   | Normal          |

**Examples:**
- Adding an optional field when compatibility flag is enabled

#### 🌶 Spicy
| Backwards Compatible | Forwards Compatible | Rollout Process |
|-----------------------|----------------------|-----------------|
| ✅                    | ❌                   | ⚠️ Staged       |

**Examples:**
- [Adding a new allowed type](./allowed-types-rollout.mdx)

#### 👮 Go to Jail
| Backwards Compatible | Forwards Compatible | Rollout Process |
|-----------------------|----------------------|-----------------|
| ❌                    | ❌                   | 🚫 Forbidden    |

**Examples:**
- Adding/removing a required field
- Removing an allowed type from a field
- Renaming a node identifier


### Checking Compatibility

Use [`TreeView.compatibility`](../../../api/fluid-framework/treeview-interface#compatibility-propertysignature) to check schema compatibility at runtime.

The two most important properties are [`canView`](../../../api/fluid-framework/schemacompatibilitystatus-interface#canview-propertysignature) (the view schema is [compatible](./index.mdx#schema-compatibility) with the stored schema) and [`canUpgrade`](../../../api/fluid-framework/schemacompatibilitystatus-interface#canupgrade-propertysignature) (it is valid to [upgrade the schema](./index.mdx#schema-upgrade-process)).

### Monitoring Remote Schema Changes

In collaborative applications, other clients may upgrade the schema while you're working.
Listen for `schemaChanged` to respond:

```typescript
// Monitor for remote schema changes
view.events.on("schemaChanged", () => {
    // Check view.compatibility here and handle accordingly
});
```

## Schema Upgrade Process

After making a [backwards compatible](./index.mdx#understanding-compatibility) schema change, upgrade existing documents using [`view.upgradeSchema()`](../../../api/fluid-framework/treeview-interface#upgradeschema-methodsignature):

```typescript
import { SchemaFactory, TreeViewConfiguration } from "@fluidframework/tree";

const factory = new SchemaFactory("WhiteboardApp");

// Updated schema with new optional field
class Note extends factory.object("Note", {
    id: factory.string,
    x: factory.number,
    y: factory.number,
    color: factory.optional(factory.string), // New optional field
}) {}

// Create tree view with updated schema
const config = new TreeViewConfiguration({ schema: Note });
const view = tree.viewWith(config);

// Check compatibility and upgrade if needed
if (!view.compatibility.canView) {
    if (view.compatibility.canUpgrade) {
        view.upgradeSchema(); // Upgrade document to new schema
        console.log("Document upgraded successfully");
    } else {
        throw new Error("Schema is incompatible and cannot be upgraded");
    }
}

// Now you can use the upgraded document
view.root.color = "#FF0000";
```

For detailed information about which types of changes are safe versus breaking, see [Types of Schema Changes](./types-of-changes.mdx).

### Staged Rollouts

Use staged rollouts for schema changes that are [backwards compatible](./index.mdx#backwards-compatibility) but not [forwards compatible](./index.mdx#forwards-compatibility). If one client upgrades a document to a schema older clients can't read, those users are locked out until they update.

**Two-phase deployment strategy**

**Phase 1: Deploy reading support**
- Deploy application versions that can _read_ the new schema but do not upgrade documents yet.
- Monitor until all (or sufficiently most) users are on the new version — this is called **client saturation**.

**Phase 2: Enable upgrades**
- Deploy code that upgrades documents to the new schema.
- Monitor for compatibility issues.

Changes that are both backwards and forwards compatible don't require staged rollouts. However, many seemingly minor changes (e.g., adding a new allowed node type) are not forwards compatible and require a staged rollout.

**Best practices:** Allow sufficient time between releases for version adoption. Monitor client version distribution before enabling upgrades. Test with mixed client versions.
