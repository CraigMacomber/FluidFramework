Understanding which schema changes are safe versus breaking is essential for evolving [SharedTree](../index.mdx) schemas without disrupting existing applications. Use a test with [`snapshotSchemaCompatibility`](../../../api/fluid-framework/#snapshotschemacompatibility-function) to verify compatibility requirements. The details below are needed to design schemas for future-compatible evolution and to interpret errors from that test.

All the examples below are modifying this schema:
```typescript
class Note extends factory.object("Note", {
    id: factory.string,
    text: factory.string,
    color: factory.optional([factory.string, factory.number]),
}) {}
```

## Compatible Changes

The following changes are [backwards compatible](./index.mdx#backwards-compatibility). Review each to determine whether it is also [forwards compatible](./index.mdx#forwards-compatibility) or requires a [staged rollout](./index.mdx#staged-rollouts).

### Adding Optional Fields

BC | FC
---| ---
✅ | ❌

```typescript
class Note extends factory.object("Note", {
    id: factory.string,
    text: factory.string,
    color: factory.optional([factory.string, factory.number]),
    height: factory.optional(factory.number), // New optional field
}) {}
```

For [forwards compatibility](./index.mdx#understanding-compatibility), set [`allowUnknownOptionalFields`](../../../api/fluid-framework/objectschemaoptions-interface.md#allowunknownoptionalfields-propertysignature) on existing clients _before_ adding the new field. If not already set, the change is forwards-incompatible and requires a [staged rollout](./index.mdx#staged-rollouts) to deploy `allowUnknownOptionalFields` first.

```typescript
class Note extends factory.object("Note", {
    id: factory.string,
    text: factory.string,
    color: factory.optional([factory.string, factory.number]),
    // Existing optional fields
}, { allowUnknownOptionalFields: true }) {}
```

### Making Required Fields Optional

BC | FC
---| ---
✅ | ❌

There is currently no way to make this change [forwards compatible](./index.mdx#forwards-compatibility).

```typescript
class Note extends factory.object("Note", {
    id: factory.string,
    text: factory.optional(factory.string), // previously required
    color: factory.optional([factory.string, factory.number]),
}) {}
```

### Adding New Allowed Types

BC | FC
---| ---
✅ | ❌

```typescript
class Note extends factory.object("Note", {
    id: factory.string,
    text: [factory.string, factory.array(factory.string)],
    color: factory.optional([factory.string, factory.number]),
}) {}
```

To achieve [forwards compatibility](./index.mdx#forwards-compatibility), use staged allowed types to [roll out](./index.mdx#staged-rollouts) read support before write support, so older clients can handle data created with the new type.
See [Rolling Out New Allowed Types](./allowed-types-rollout.mdx) for details.

## Incompatible Changes

These changes break backwards compatibility, forwards compatibility, or both — preventing some client versions from opening some document versions.

### Removing Fields

```typescript
// Breaking: Removing field
class Note extends factory.object("Note", {
    id: factory.string,
    // text: factory.string, // REMOVED
    color: factory.optional([factory.string, factory.number]),
}) {}
```

Backward compatibility: newer clients cannot open old documents, which contain a `text` field they don't know how to read or write.

Forward compatibility: older clients cannot open new documents, which are missing the `text` field they expect.

### Making Optional Fields Required

```typescript
// Breaking: Making optional field required
class Note extends factory.object("Note", {
    id: factory.string,
    text: factory.string,
    color: [factory.string, factory.number], // Was optional, now required - BREAKING!
}) {}
```

Backward compatibility: newer clients cannot open old documents, which may lack the `color` field that is now required.

Forward compatibility: older clients cannot open new documents, which require `color` to be set — something older clients may not do and cannot clear.

### Changing Field Types

```typescript
// Breaking: Changing field type
class Note extends factory.object("Note", {
    id: factory.number, // Was factory.string - BREAKING!
    text: factory.string,
    color: factory.optional([factory.string, factory.number]),
}) {}
```

Backward compatibility: newer clients cannot open old documents, which store `id` as `string` while the new schema treats it as `number`.

Forward compatibility: older clients cannot open new documents, which store `id` as `number` while the old schema treats it as `string`.

### Removing Allowed Types

```typescript
// Breaking: Removing allowed type
class Note extends factory.object("Note", {
    id: factory.string,
    text: factory.string,
    color: factory.optional([factory.string]), // removed factory.number
}) {}
```

Backward compatibility: newer clients cannot open old documents, which may store `color` as `number` — a type the new schema no longer allows.

Forward compatibility: older clients cannot open new documents, which only store `color` as `string`, but older clients expected to also write it as `number`.
